# Provenance Guard

Provenance Guard is a backend classification API that evaluates text-based content to determine if it was written by a human or generated by an AI. It returns a dynamic transparency label, maintains an SQLite audit log of decisions, and provides an appeals endpoint for creators to contest classifications.

## Architecture Overview
The path of a single submission flows as follows:
1. **Submit**: A client sends a `POST` request to `/submit` with the text and creator ID.
2. **Signal Analysis**: The text is immediately evaluated by two distinct signals:
   - **Groq LLM**: A semantic analysis using `llama-3.3-70b-versatile`.
   - **Stylometrics**: A structural heuristic computing type-token ratio and sentence variance.
3. **Scoring & Labeling**: The two signal scores are averaged into a final confidence score, which maps to one of three dynamic transparency labels.
4. **Audit Log**: The individual scores, final confidence score, and classification status are inserted into an SQLite database (`audit_log` table).
5. **Response**: The API returns the unique `content_id`, confidence score, and transparency label to the client.

If a creator wishes to appeal, they hit the `POST /appeal` endpoint with their `content_id` and reasoning. The system locates the record in the audit log and updates its status to `under_review`.

## Detection Signals
1. **LLM-Based Classification (Groq)**:
   - **What it measures**: Evaluates the semantic coherence, tone, and stylistic choices of the text.
   - **Why it was chosen**: LLMs are currently the best at holistically recognizing "AI tone" (predictable transitions, lack of genuine emotional depth).
   - **What it misses**: May struggle with highly formal or academic human writing that inherently lacks emotional variation, or heavily prompted AI output designed to mimic human flaws.

2. **Stylometric Heuristics (Python)**:
   - **What it measures**: Analyzes structural properties of the text such as sentence length variance and vocabulary diversity (type-token ratio). 
   - **Why it was chosen**: AI writing tends to be structurally uniform, whereas human writing varies more naturally. It provides a purely mathematical counter-balance to the LLM's semantic opinion.
   - **What it misses**: Can be fooled by AI text specifically prompted to vary sentence lengths, or human writing that is naturally very repetitive or short.

## Confidence Scoring
The two signals each output a score from 0.0 (Human) to 1.0 (AI). The system averages these two scores to produce a final `confidence` score. 
Validating that the score is meaningful involves passing both completely uniform AI text (which produces a low stylometric variance and high LLM score) and casual, choppy human text (which produces high variance and a low LLM score). 

**Example Submissions:**
1. **High-Confidence Human (Score: 0.2)**: 
   *Input*: "The sun dipped below the horizon, painting the sky in hues of amber and rose. I sat on the porch, coffee in hand, watching the neighborhood slowly go quiet." 
   *Result*: Both signals agreed this was highly variable and human-like.
2. **Uncertain / Mixed (Score: 0.5)**: 
   *Input*: A short, generic 2-sentence formal statement. 
   *Result*: The text was too short to build a reliable sentence length variance (Stylometrics score ~0.8), while the LLM gave it a ~0.2, resulting in an uncertain average of 0.5.

## Transparency Labels
Based on the final confidence score, the user will see one of the following exact labels:

*   **High-Confidence Human (0.0 - 0.35)**: `"This content was likely written by a human."`
*   **Uncertain (0.36 - 0.64)**: `"The origin of this content is unclear; it may contain a mix of human and AI generation."`
*   **High-Confidence AI (0.65 - 1.0)**: `"This content was likely generated by AI."`

## Rate Limiting
The `/submit` endpoint is protected by `Flask-Limiter` using in-memory storage, capped at:
*   **10 per minute**: Allows a creator to upload a batch of work quickly, but stops brute-force bot flooding.
*   **100 per day**: A realistic upper bound for an active writer on a platform in a 24-hour period.

## Known Limitations
*   **Short, Casual Texts**: Because our Stylometrics signal relies heavily on sentence length variance and vocabulary diversity, passing it a 1 or 2 sentence casual post will break the math (low variance), causing it to artificially spike towards 1.0 (AI). The system currently defaults very short texts to 0.5 to mitigate this, but it struggles to confidently classify brief snippets.

## Spec Reflection
*   **How it helped**: Designing the `planning.md` architecture diagram and data flow explicitly before coding made wiring up the `insert_log()` function effortless because I already knew exactly which fields needed to exist.
*   **How implementation diverged**: During implementation, I realized computing complex Type-Token ratios for short texts was too volatile. I diverged from the spec slightly by heavily relying on sentence length variance instead, and hardcoding a fallback `0.5` score for texts under 15 words to prevent false positives.

## AI Usage
1. **Flask Skeleton & Signal 1**: I directed the AI assistant to generate the initial Flask `app.py` routing structure and the Groq LLM API call based heavily on my `planning.md` spec. I had to manually intervene and override the code when the sandbox prevented it from executing `pip install` background processes.
2. **Stylometrics Implementation**: I instructed the AI to write a regular expression-based sentence parser in Python to calculate variance. I reviewed the code to ensure it was properly handling empty strings and edge cases (like zero-division errors when there is only 1 sentence).

## Stretch Features Implemented
1. **Provenance Certificate (`/verify`)**: Added a credential system where creators can be verified. Verified creators receive a `verified_human: true` flag in their JSON response, and their Transparency Label is appended with "🏅 Verified Human".
2. **Analytics Dashboard (`/analytics`)**: Added an endpoint that runs SQLite aggregations to return platform-wide statistics, including detection patterns, appeal rates, and the average confidence score across all submissions.
3. **Multi-modal Support (Metadata)**: Extended the `/submit` endpoint to accept an optional `metadata` dictionary (e.g., EXIF camera data). This metadata is dynamically embedded into the prompt for the Groq LLM to cross-reference against the text for inconsistencies.
